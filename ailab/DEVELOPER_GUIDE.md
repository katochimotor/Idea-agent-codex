# Developer Guide

This guide explains how the current AI Idea Research Lab implementation works and how to extend it safely.

## 1. What This System Is

AI Idea Research Lab is a local-first Windows application for idea discovery.

Current runtime shape:

- `start.bat` in the repository root launches the app for end users.
- `launcher/launcher.py` starts backend and frontend, then opens the browser.
- `backend/` provides the API, job worker, orchestration, database access, and local vector search.
- `frontend/` provides the dashboard UI.
- `SQLite` stores metadata and relationships.
- `data/vector_index/` stores the local on-disk vector index.
- `idea_reports/` and `projects/` store generated output files.

The app is intentionally simple in deployment:

- one machine
- one local database
- one local background worker
- one local vector index

## 2. Main Runtime Flow

### Startup

1. User double-clicks `start.bat`.
2. `start.bat` runs `ailab\.venv\Scripts\python.exe launcher\launcher.py`.
3. `launcher.py` starts:
   - backend via Uvicorn
   - frontend via Vite
4. Backend startup:
   - applies DB migrations
   - seeds sample data if the DB is empty
   - starts the background worker thread
5. Browser opens the frontend on `http://127.0.0.1:3000`.

### Idea discovery

There are now two execution paths:

- synchronous compatibility path
  - the existing UI still calls discovery and waits for the result
  - internally it now uses the same orchestration service as background jobs
- asynchronous job path
  - frontend creates a job through `/api/jobs/discover`
  - frontend polls `/api/jobs/{job_id}`
  - worker claims and executes the job
  - pipeline persists ideas, scores, reports, and vector updates

### Search

The local search flow is:

1. documents are chunked into `document_chunks`
2. chunk vectors are generated by the local embedding service
3. vector metadata is stored in SQLite
4. actual vectors are written to `data/vector_index/*.json`
5. `/api/search/documents` performs nearest-neighbor lookup over the local index

## 4. Backend Architecture

### 4.1 API layer

The FastAPI app is assembled in `backend/app.py`.

Responsibilities:

- register routers
- configure CORS
- initialize DB and seed data
- start and stop the worker thread

### 4.2 Controllers

Controllers are thin adapters used by HTTP routes.

Examples:

- `IdeaController`
  - list ideas
  - read idea details
  - invoke discovery orchestration for the sync compatibility path
- `ProjectController`
  - create a starter project directory and DB records
- `AnalysisController`
  - return dashboard metrics
- `DashboardController`
  - return static dashboard overview content

The system is moving toward service-driven orchestration. New complex behavior should usually go into a service, not a controller.

### 4.3 Orchestration service

`backend/services/pipeline_orchestration_service.py` is the center of the backend.

Responsibilities:

- create `pipeline_runs`
- resolve models from the model registry
- run the idea pipeline
- deduplicate generated ideas
- create documents and ideas
- write idea scores and reports
- rebuild the vector index after discovery
- execute supported background job types

This is the main place to extend when you add:

- new pipeline stages
- new job types
- new persistence side effects
- new indexing behavior

### 4.4 Job system

The background worker is intentionally local and small.

Components:

- `JobRepository`
  - low-level DB operations for jobs and events
- `JobService`
  - enqueue, read, execute jobs
- `BackgroundJobWorker`
  - loop that claims queued jobs and runs them

Current job types:

- `discover_ideas`
- `refresh_embeddings`

Worker model:

- background daemon thread inside the FastAPI process
- polling loop using SQLite
- single-worker semantics

This is acceptable for the current MVP but should eventually become a separate process if concurrency or reliability requirements increase.

## 6. Search and Embeddings

The current search system is intentionally local and deterministic.

Components:

- `backend/search/embedding_service.py`
  - lightweight hash-based embedding function
- `backend/search/vector_index.py`
  - JSON-backed local vector store
- `backend/search/retriever.py`
  - chunking, indexing, collection management, querying

Current indexed entity:

- `document_chunks`

Current search endpoint:

- `POST /api/search/documents`

Important design detail:

- vectors themselves live on disk
- SQLite stores vector metadata and entity mapping

That separation makes it easier to replace the local vector backend later.

## 8. How Data Moves Through the System

### Discovery path

1. Frontend calls `/api/jobs/discover` or the sync discovery wrapper.
2. A `jobs` row is created.
3. Worker claims the job and marks it `running`.
4. `PipelineOrchestrationService` creates a `pipeline_runs` row.
5. `IdeaPipeline` generates candidate ideas from collected research inputs.
6. Semantic deduplication filters near-duplicates.
7. New rows are written to:
   - `documents`
   - `ideas`
   - `idea_scores`
   - `reports`
8. Vector indexing updates:
   - `document_chunks`
   - `vector_collections`
   - `vector_entries`
   - `vector_sync_runs`
9. Job status becomes `completed`.
10. Frontend polling sees completion and refreshes the idea list.

### Search path

1. API receives a query.
2. Query text is embedded.
3. Local vector index returns nearest chunk keys.
4. Keys are mapped back to `vector_entries`, then `document_chunks`, then `documents`.
5. API returns the ranked evidence snippets.

## 10. How To Extend The System

### 10.1 Add a new job type

1. Decide the job payload shape.
2. Enqueue it through `JobService`.
3. Implement execution in `PipelineOrchestrationService.execute_job`.
4. If needed, create a dedicated orchestration method similar to:
   - `discover_and_persist`
   - `rebuild_vector_index`
5. Expose an API route in `backend/api/routes_jobs.py`.
6. Add frontend polling if the UI needs to wait for completion.

Rule:

- do not put long-running logic directly in route handlers

### 10.2 Add a new pipeline stage

1. Add or update an agent in `backend/agents/`.
2. Wire it into the relevant pipeline in `backend/pipelines/`.
3. Update `PipelineOrchestrationService` if the new stage affects persistence, indexing, or run metadata.
4. If the stage needs traceability, write its outputs to a dedicated table and connect it to `pipeline_runs`.

Examples:

- problem extraction persistence
- cluster persistence
- idea evidence generation

### 10.3 Add a new source connector

The current scaffold has source services, but the target shape is a full ingestion framework.

Current simple path:

1. Add a service or connector module for the new source.
2. Return normalized discussion items with source metadata.
3. Plug it into collection logic used by the pipeline.
4. Create a `sources` row for the new source key.

Recommended future path:

1. add a connector under `backend/ingestion/connectors/`
2. implement fetch, normalize, fingerprint, metadata extraction
3. store checkpoints in `source_checkpoints`

### 10.4 Add a new search entity

To index something beyond `document_chunks`:

1. add a collection in `vector_collections`
2. generate vectors for the new entity type
3. persist mapping rows in `vector_entries`
4. extend `VectorSearchService` with:
   - build method
   - query method
5. add a new API endpoint in `routes_search.py`

Examples:

- `problems`
- `problem_clusters`
- `ideas`

### 10.5 Replace the local embedding model

Current embeddings are local and deterministic.

To switch to a real model:

1. replace or wrap `LocalEmbeddingService`
2. add or update the corresponding `model_registry_entries` row
3. keep the `VectorSearchService` public interface unchanged
4. rebuild the vector index

This is intentionally isolated so the search API does not need to change.

### 10.6 Add a new AI model or provider

1. add a `model_registry_entries` row
2. set `task_key`
3. mark one row as `is_default=1` for that task if appropriate
4. update the model-resolving service or AI client if provider-specific behavior is needed
5. record usage through `pipeline_runs`

The important contract is:

- every meaningful AI-generated artifact should be traceable to a model entry and run

### 10.7 Extend the frontend

When adding UI features:

1. keep API code in `frontend/src/api/`
2. keep page-specific state in `frontend/src/pages/`
3. keep reusable visual units in `frontend/src/components/`
4. prefer adding new endpoints instead of overloading existing ones

Examples:

- a job status panel can poll `/api/jobs/{id}/events`
- a search page can call `/api/search/documents`
- an evidence drawer can show search results linked to an idea

## 11. Safe Change Rules

Follow these rules when extending the system:

- keep `schema.sql` as the DB source of truth
- keep route handlers thin
- keep orchestration in services
- keep long-running work in jobs
- keep AI outputs linked to `pipeline_runs`
- keep vector data mapped through `vector_entries`
- keep user-facing text in Russian
- do not bypass the migration path by manually changing the DB

## 12. Current Limitations

The current implementation is still an MVP scaffold.

Known limitations:

- one in-process worker thread
- one local JSON vector index
- local hash embeddings, not semantic model embeddings
- only document chunk search is implemented
- clustering and extraction persistence are not fully wired yet
- no WebSocket progress streaming
- no auth or multi-user isolation

These are intentional tradeoffs to keep the system runnable and easy to extend.

## 13. Recommended Next Extensions

If you want to evolve the system in a clean order, do this next:

1. persist extracted problems and problem clusters
2. index `problems`, `clusters`, and `ideas`
3. add source checkpoints and a real ingestion connector framework
4. add `pipeline_runs` views to the frontend
5. replace local embeddings with a configurable real embedding model
6. move the worker into a separate process if job volume grows

## 14. Quick Debug Checklist

If the app starts but discovery fails:

- check `backend/logs/ailab.log`
- check `/health`
- check `/api/jobs/{job_id}`
- check `/api/jobs/{job_id}/events`
- check that `data/vector_index/` exists and is writable

If search returns nothing:

- rebuild embeddings through `/api/jobs/embeddings/rebuild`
- confirm `document_chunks` rows exist
- confirm `vector_entries` rows exist
- confirm the index file exists in `data/vector_index/`

If the frontend does not refresh after discovery:

- inspect `/api/jobs/{job_id}`
- confirm the job reaches `completed`
- confirm `/api/ideas` shows new rows

This guide should be updated whenever the worker model, persistence layout, or search layer changes.
